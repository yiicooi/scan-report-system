## 数据库模型详细设计

### RBAC 权限相关

```
users
  id, name, password_hash, department_id, role_id, is_active, created_at

departments
  id, name, parent_id（支持多级）, created_at

roles
  id, name, description

permissions
  id, resource, action        -- 例: resource=order, action=create

role_permissions
  role_id, permission_id
```

### 工艺相关

```
processes（工序模板）
  id, name, department_id, created_at
  -- 只存名称，不存工时（工时因零件而异）

process_aliases（流程模板/别名）
  id, alias_name, department_id, note, created_at
  -- 一个别名代表一组有序工序配置

process_alias_items（流程模板明细）
  id, alias_id, process_id, display_name（显示别名）, sort
  -- 导入时填入工序名和排序，工时仍需文员手动录入
```

### 工单相关

```
orders（工单）
  id
  internal_no（内部单号，唯一，用于生成二维码）
             -- 主单格式：SC_2026050001
             -- 报废单格式：SC_2026050001_BF01（BF02 表示第二张报废单）
  parent_id（关联主订单 id，主单为 NULL，报废单指向主单）
  order_type  -- normal（主单）/ scrap（报废单）
  external_no（外部单号）
  drawing_no（图纸编号）
  drawing_url（阿里云 OSS URL，可为空）
  total_qty（订单总数量）
  unit_price（单价）
  total_amount（总额）
  order_date（订单日期）
  status  -- draft / active / completed
           -- 主单：所有关联报废单也 completed 才能置 completed
  total_completed（冗余字段：最后一道工序的 total_completed，避免 JOIN）
  created_by（user_id）
  created_at, updated_at

order_processes（工单工序明细）
  id, order_id, process_id
  display_name（本工单该工序的显示名称）
  sort（排序，决定工序先后）
  unit_hours（单个工时，每道工序单独设置，不存于工序模板）
  total_hours（总工时，可手动覆盖）
  deadline（最后完成时间）
  status  -- pending / in_progress / completed
  created_at, updated_at
```

### 报工相关（汇总表 + 明细表）

```
work_report_summaries（报工汇总表）
  id, order_process_id（唯一索引）
  total_received（该工序累计接收）
  total_completed（该工序累计完成）
  total_scrap（该工序累计报废）
  progress_pct（进度百分比，0.00~100.00，= total_completed / order.total_qty * 100，随事务更新）
  is_completed（bool）
  updated_at
  -- 每道工序恰好一条，随每次报工事务更新

work_report_details（报工明细表）
  id
  summary_id
  order_process_id
  user_id
  received_qty（本次接收，可为 0）
  completed_qty（本次完成，可为 0）
  scrap_qty（本次报废，可为 0）
  receive_images（JSON 数组，OSS URL）
  complete_images（JSON 数组，OSS URL）
  scrap_images（JSON 数组，OSS URL）
  note（备注）
  reported_at
  -- 约束：三个数量不能同时为 0
```

**WorkReportDetail 设计说明：一行多列**

- 一次提交 = 一条记录，含 `received_qty / completed_qty / scrap_qty` 各一列
- 数量为 0 表示本次未操作该项，语义清晰
- 避免多条+type 方案中 `batch_id` 关联的复杂度
- 查询某次完整操作只需读一行，JOIN 简单
- 工人一次操作天然对应一条记录

## 报工业务规则与事务流程

### 数量约束

```
新的 total_received  = 当前 total_received  + 本次 received_qty
新的 total_completed = 当前 total_completed + 本次 completed_qty

约束：
  新 total_received  ≤ order.total_qty
  新 total_completed ≤ 新 total_received
  scrap_qty ≥ 0（报废只记录，不参与约束链）
  三个数量不能同时为 0
```

### 事务流程（service/report.go）

```
BEGIN TRANSACTION

  1. SELECT work_report_summaries WHERE order_process_id = ? FOR UPDATE
     -- 行锁，防止并发超量

  2. 数量校验（如超量 → ROLLBACK，返回 422）

  3. INSERT work_report_details（本次明细，一行多列）

  4. UPDATE work_report_summaries SET
       total_received  += received_qty,
       total_completed += completed_qty,
       total_scrap     += scrap_qty,
       progress_pct     = 新 total_completed / order.total_qty * 100,
       is_completed     = (新 total_completed >= order.total_qty)

  5. 若首次有数量 → UPDATE order_processes SET status = in_progress

  6. 若 is_completed = true → UPDATE order_processes SET status = completed

  7. 若该工序是最后一道（sort 最大）
       → UPDATE orders SET total_completed = 新 total_completed
         （最后工序只同步完成数量，不做 >= total_qty 约束，
           因为报废单会分担部分数量）

  8. 订单完成判断（仅在最后工序 is_completed 时触发）
       若 order.order_type = normal：
         查询所有关联报废单（parent_id = order.id）
         判断：主单 total_completed + SUM(报废单 total_completed) >= order.total_qty
         AND   所有关联报废单 status = completed
         若满足 → UPDATE orders SET status = completed（主单）
       若 order.order_type = scrap：
         同步通知主单重新执行上述判断

COMMIT
```

### 工序状态流转

```
pending ──(首次报工)──► in_progress ──(累计完成 ≥ 本单 total_qty)──► completed
```

### 订单完成判断（主单 + 报废单联动）

```
主单 completed 条件：
  主单最后工序 is_completed = true
  AND 主单 total_completed + Σ(所有报废单 total_completed) >= 主单 total_qty
  AND 所有关联报废单 status = completed

内部单号规则：
  主单：SC_YYYYMM0001  （如 SC_2026050001）
  报废单：SC_2026050001_BF01 / _BF02 ...（顺序编号）
```

## GORM Migration 策略

```go
// database/migrate.go
func AutoMigrate(db *gorm.DB) error {
    return db.AutoMigrate(
        &model.User{},
        &model.Department{},
        &model.Role{},
        &model.Permission{},
        &model.RolePermission{},
        &model.Process{},
        &model.ProcessAlias{},
        &model.ProcessAliasItem{},
        &model.Order{},
        &model.OrderProcess{},
        &model.WorkReportSummary{},
        &model.WorkReportDetail{},
    )
}
```

- **新增字段**：AutoMigrate 自动处理
- **索引 / 约束 / 重命名列**：`migrations/` 目录下 SQL 文件，用 `golang-migrate` 按版本执行

### 扫码报工页面完整流程

```
1. 点击"二维码扫描"
   └─ mobile_scanner 扫描 internal_no
   └─ 调用 GET /api/app/scan?internal_no=xxx
   └─ 自动填充：零件名称 / 图纸编号 / 内部单号 / 外部单号 / 工艺流程 / 订单日期

2. 下拉选择报工工序
   └─ 列表展示该工单所有工序
   └─ 标注状态：待开始 / 进行中🟡 / 已完成✅
   └─ 显示工序汇总：订单数量 / 我已完成 / 上道工序完成

3. 填写本次报工数量
   └─ 本次接收（可为 0）+ 接收图片（可选）
   └─ 本次完成（可为 0）+ 完成图片（可选）
   └─ 本次报废（可为 0）+ 报废图片（可选）
   └─ 三者不能同时为 0

4. 本地预校验（提交前）
   └─ 汇总接收 + 本次接收 ≤ 订单数量
   └─ 汇总完成 + 本次完成 ≤ 汇总接收 + 本次接收

5. 图片上传（如有）
   └─ 调用 POST /api/app/oss/presign 获取预签名 URL
   └─ 直传阿里云 OSS，拿到图片 URL

6. 提交 POST /api/app/report
   └─ 后端事务校验 + 写库
   └─ 成功 → Toast 提示 + 刷新工序汇总数据
   └─ 失败（越界）→ 显示具体错误原因
```

## 验证计划

| 场景 | 验证方式 |
|------|---------|
| 接收超出订单数量 | API 返回 422，汇总不变 |
| 完成超出接收数量 | API 返回 422 |
| 两工人并发报工同一工序 | SELECT FOR UPDATE 保证串行，不超量 |
| 首次报工触发状态变更 | 工序 status → in_progress |
| 完成量达标触发工序完成 | 工序 status → completed |
| 最后工序完成同步订单 | Order.total_completed 更新，不独立判断 >= total_qty |
| 主单+报废单共同完成 | 主单完成数 + 所有报废单完成数 >= total_qty 且报废单全完成 |
| 报废单编号生成 | 自动取主单现有报废单数量 +1，生成 _BF01/_BF02 |
| 进度百分比 | progress_pct = total_completed / total_qty * 100，随每次报工更新 |
| 图纸上传为空 | drawing_url 为 null，正常录单 |
| 流程模板导入工时仍需手填 | 前端工时列不自动填充，强制输入 |
